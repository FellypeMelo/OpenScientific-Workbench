# Seção 1 – Glossário Formal (Layer 13 - Evolução e Gestão de Configuração)

**ID:** ARCH-RULESET-L13-EVOL-GLOSSARY  
**Status:** Definitivo  
**Escopo:** Definições canônicas obrigatórias de terminologias de evolução de software e gerência de configuração segundo Sommerville.

---

## Termos Canônicos

| Termo | Definição Canônica (Obrigatória) | Fonte |
| :--- | :--- | :--- |
| **Evolução de Software** | Processo contínuo de mudanças em um sistema de software após sua entrega inicial, para corrigir defeitos, adaptar-se a novos ambientes e atender a novos requisitos. | Cap. 9 |
| **Manutenção Corretiva** | Tipo de manutenção focada na correção de defeitos (bugs) identificados durante a operação do sistema. | Cap. 9.3 |
| **Manutenção Adaptativa** | Tipo de manutenção focada na adaptação do sistema a mudanças em seu ambiente operacional (hardware, sistema operacional, plataformas). | Cap. 9.3 |
| **Manutenção Perfectiva** | Tipo de manutenção focada na melhoria do sistema, adicionando novas funcionalidades ou melhorando a performance e a estrutura. | Cap. 9.3 |
| **Leis de Lehman** | Conjunto de leis empíricas sobre a evolução de sistemas de software, incluindo: (1) Mudança Contínua; (2) Complexidade Crescente; (3) Evolução como Processo Auto-Regulador; (4) Estabilidade Organizacional; (5) Conservação da Familiaridade; (6) Crescimento Contínuo; (7) Qualidade Declinante; (8) Sistema de Feedback. | Cap. 9.2 |
| **Reengenharia (Reengineering)** | Processo de modificação de um sistema existente para melhorar sua estrutura, legibilidade e manutenibilidade, sem alterar sua funcionalidade externa. | Cap. 9.3.2 |
| **Refatoração (Refactoring)** | Melhoria contínua da estrutura interna do código sem alterar seu comportamento externo, visando reduzir dívida técnica e facilitar mudanças futuras. | Cap. 9.3.3 |
| **Sistema Legado** | Sistema antigo que ainda é útil e crítico para o negócio, mas que foi desenvolvido com tecnologias obsoletas e cuja manutenção é cara e arriscada. | Cap. 9.4 |
| **Gestão de Configuração (Configuration Management – CM)** | Processo de gerenciar mudanças em um sistema de software, incluindo versionamento, controle de mudanças, build e release. | Cap. 25 |
| **Versionamento (Version Control)** | Processo de rastrear e gerenciar diferentes versões de artefatos de software (código, documentação, configurações). | Cap. 25.2 |
| **Branch (Ramo)** | Linha de desenvolvimento paralela a partir de uma versão base, permitindo desenvolvimento isolado de funcionalidades ou correções. | Cap. 25.2 |
| **Merge (Mesclagem)** | Processo de combinar alterações de diferentes branches em uma única versão. | Cap. 25.2 |
| **Build** | Processo de compilar, linkar e empacotar componentes de software para criar uma versão executável. | Cap. 25.3 |
| **Release** | Versão de um sistema que é disponibilizada para usuários ou clientes. | Cap. 25.4 |
| **Baseline** | Conjunto de versões de componentes que define uma versão específica do sistema, congelada para referência futura. | Cap. 25.2 |
| **Change Request (CR)** | Solicitação formal de mudança em um sistema, contendo descrição, justificativa, impacto estimado e prioridade. | Cap. 25.1 |
