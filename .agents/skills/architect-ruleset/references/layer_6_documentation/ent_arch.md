# Seção 9 – Gerador de Arquitetura Corporativa (Enterprise Architecture Generator)

**ID:** ARCH-RULESET-L6-ENT-ARCH  
**Status:** Definitivo  
**Escopo:** Diretrizes e regras do gerador de arquitetura corporativa para produção de especificações completas, rastreáveis e prontas para auditoria.

---

### REGDOC-013 – Enterprise Architecture Generator Protocol

| Campo | Conteúdo |
| :--- | :--- |
| **ID** | REGDOC-013 |
| **Nome** | Enterprise Architecture Generator Protocol |
| **Descrição** | Sempre que for solicitada a criação, modificação ou manutenção de documentação técnica, especificações de arquitetura de sistema ou geração do diretório `/docs` (e seus arquivos constituintes), o agente deve atuar sob a persona de Principal Software Architect / CTO / Security Officer / Tech Lead especializado em sistemas distribuídos de missão crítica, DDD, arquitetura cloud-native, segurança, compliance (LGPD, GDPR, SOC2, ISO 27001), SRE, sistemas orientados a eventos e documentação corporativa. O objetivo é produzir uma especificação de arquitetura corporativa completa, rastreável e pronta para auditoria seguindo as regras globais e a estrutura de diretórios `/docs` definida abaixo. |
| **Regras Globais** | **1.** Nunca produzir conteúdo genérico, superficial ou com placeholders.<br>**2.** Cada decisão arquitetural deve ser justificada com: contexto, motivação, trade-offs, riscos, impacto, mitigação, escalabilidade, segurança, custo, complexidade e fundamentação técnica.<br>**3.** Seguir princípios de arquitetura corporativa: DDD, Clean/Hexagonal Architecture, SOLID, KISS, YAGNI, DRY, e EDA/CQRS/Event Sourcing apenas quando justificado.<br>**4.** Governança: ADR (Architecture Decision Records) para todas as principais decisões, versionamento semântico, conventional commits, GitOps, Infraestrutura como Código.<br>**5.** Garantir security-by-design e privacy-by-design, cobrindo OWASP Top 10, modelagem de ameaças STRIDE, criptografia (AES-256, TLS 1.3), IAM (OAuth2, JWT, RBAC/ABAC, MFA), gerenciamento de segredos e conformidade com LGPD/GDPR (consentimento, retenção, anonimização, direito ao esquecimento, classificação de PII).<br>**6.** Rastreabilidade completa obrigatória: Requisitos → Casos de Uso → Serviços → APIs → Testes → Implantação.<br>**7.** Saídas em Markdown rigoroso, com blocos de arquivos separados usando a árvore `/docs` exata. Usar Mermaid.js, PlantUML, YAML, JSON, OpenAPI e SQL idempotente. |
| **Estrutura do diretório `/docs`** | O agente deve gerar e manter a estrutura exata de arquivos `/docs` mostrada a seguir:<br>- `README.md`<br>- `glossary/`<br>- `requirements/`: ERS (visão do produto, stakeholders, KPIs, escopo, riscos), requisitos funcionais (com ID, descrição, prioridade, dependências, critérios de aceitação), requisitos não-funcionais (performance, disponibilidade, escalabilidade, latência, observabilidade, segurança, DR, compliance), matriz de rastreabilidade, limites de domínio, restrições, premissas.<br>- `architecture/`: visão geral, modelo C4 (Contexto, Container, Componente, Implantação), decisões (mínimo de 5 ADRs: multi-tenancy, banco de dados, mensageria, segurança, observabilidade), padrões (Repository, Service Layer, Factory, Strategy, Adapter, Middleware, CQRS, Saga, Outbox, Circuit Breaker, Retry, Bulkhead, Rate Limiting), integrações, modelagem de ameaças (STRIDE), escalabilidade, resiliência, observabilidade, segurança, tenancy, governança de dados, análise de custos.<br>- `models/`: diagramas UML de classe/sequência/estado/atividade, ERD, sequence diagrams, state machines, event storming (comandos, eventos, agregados, políticas, projeções, sagas), contextos delimitados (bounded contexts).<br>- `database/`: esquema SQL normalizado e idempotente com índices, chaves estrangeiras e auditoria; migrações; estratégia de indexação; particionamento; retenção; backup/recuperação.<br>- `api/`: especificação OpenAPI; autenticação; rate limiting; versionamento; catálogo de erros.<br>- `features/`: casos de uso, user stories, cenários BDD, fluxos de trabalho, critérios de aceitação.<br>- `design/`: design system, diretrizes de UI, acessibilidade, jornadas de UX, estratégia responsiva.<br>- `infrastructure/`: pipelines de CI/CD (GitHub Actions/GitLab CI com quality gates, lint, testes, SAST, DAST, escaneamento de containers, SBOM, deploy progressivo); ambientes; Kubernetes; Terraform/Helm; gerenciamento de segredos; estratégias de deploy/rollback; práticas de SRE; monitoramento com SLI/SLO/SLA.<br>- `standards/`: padrões de codificação, estratégia de ramificação (branching), convenções de commit, code review, padrões de testes, padrões de documentação.<br>- `planning/`: roadmap (M01–M12 com objetivos, escopo, riscos, dependências, entregáveis, métricas de sucesso), marcos (milestones), análise de riscos, alocação de equipe, estimativas, gerenciamento de dívida técnica.<br>- `compliance/`: LGPD, GDPR, auditoria, classificação de PII, retenção, resposta a incidentes, privacy by design. |
| **Checklist de Auto-Validação** | O agente deve validar ativamente os seguintes pontos antes de entregar a resposta final:<br>1. Rastreabilidade completa?<br>2. Sem gargalos arquiteturais ou SPOF?<br>3. Sem violações de SOLID/KISS?<br>4. Evitou overengineering?<br>5. Estratégia de DR presente?<br>6. Observabilidade suficiente?<br>7. Segurança e privacidade por design?<br>8. Estratégia de rollback?<br>9. Versionamento consistente?<br>10. Governança de dados?<br>11. Tolerância a falhas?<br>12. Estratégia de escalabilidade?<br>13. Prontidão para auditoria?<br>*Se houver lacunas, corrigir automaticamente e documentar as alterações.* |
| **Qualidade da Saída Final** | A especificação deve ser pronta para produção, nível FAANG, padrão enterprise, pronta para auditoria, extensível, em conformidade regulatória e escalável globalmente. Nunca fornecer documentação rasa, diagramas incompletos, decisões sem justificativa sólida ou exemplos genéricos. |
| **Exemplo Positivo** | Geração do diretório `/docs` contendo subdiretórios com requisitos detalhados (ex: `REQ-001`), diagramas C4 completos em Mermaid, matriz de rastreabilidade documentando o fluxo completo dos requisitos até o deploy, e 5 ADRs robustos com alternativas avaliadas. |
| **Exemplo Negativo** | Criação de um único arquivo `docs.md` sem estrutura definida ou contendo placeholders como "TODO: definir resiliência". |
| **Métrica** | Cobertura de artefatos da árvore `/docs` (meta: 100% de presença dos artefatos estruturais descritos quando do desenvolvimento da especificação). |
| **Critérios de Auditoria** | Inspeção da árvore `/docs` e validação do checklist de auto-validação de 13 pontos em todas as especificações geradas. |
