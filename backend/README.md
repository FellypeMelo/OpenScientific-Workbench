# osw-backend

Backend Core for OpenScientific-Workbench: a Clean Architecture / DDD FastAPI
gateway coordinating remote bioinformatics workflows (agent sessions, workspace
forking, HPC/Slurm dispatch, GraphRAG, and LLM streaming).

This file exists primarily as packaging metadata: `pyproject.toml` declares
`readme = "README.md"`, which setuptools requires to be present to build the
project's metadata (`pip install .`, used by `backend/Dockerfile`). See the
root [`README.md`](../README.md) for the full architectural rules and the
package layout under `src/` (`domain/`, `application/`, `infrastructure/`,
`presentation/`).

Local development: see `.env.example` for configuration and
`docs/planning/execution_plan_gap_closure.md` for the phased delivery history.

## Scientific tool catalog / sandbox toolkit

`backend/Dockerfile`'s `sandbox-toolkit` build stage installs a dedicated conda/micromamba
environment (`backend/sandbox/environment.yml` + `requirements-pip.txt` -- biopython, RDKit,
samtools, R/Bioconductor, ~150 packages total) that sandboxed action tools run against inside the
`bwrap` jail (`src/infrastructure/sandbox/bubblewrap_driver.py`), kept separate from this app's own
`/opt/venv` on purpose. See **[`docs/tools/README.md`](docs/tools/README.md)** for the full 172-tool
catalog, and `data_lake/MANIFEST.md` (repo root) + `scripts/fetch_data_lake.py` for the bundled
reference datasets some tools read.

## Database schema migrations (Alembic)

Schema changes are managed with [Alembic](https://alembic.sqlalchemy.org/) under
`backend/migrations/`, targeting the SQLAlchemy models in
`src/infrastructure/persistence/models.py`. `migrations/env.py` reads the DB URL
from the same `settings.DATABASE_URL` the app itself uses (`src/infrastructure/config.py`),
so no separate connection string needs to be maintained.

**Local dev / test**: nothing to do -- `presentation/main.py`'s `lifespan` calls
`Base.metadata.create_all` automatically on startup whenever `settings.ENV !=
"production"`, so a fresh SQLite/Postgres database is bootstrapped with zero setup.

**Production**: `create_all` is skipped entirely when `ENV=production` (see
`lifespan` in `src/presentation/main.py`). The deployment process MUST run the
migration as a separate init step, against the target Postgres database, BEFORE
any app replica starts serving traffic:

```bash
cd backend
alembic upgrade head
```

Or, against the built container image (same image the root
`docker-compose.yml`'s `backend` service runs), before starting the stack:

```bash
docker compose run --rm backend alembic upgrade head
```

**Generating a new migration** after changing `models.py`:

```bash
cd backend
alembic revision --autogenerate -m "describe the schema change"
```

Always review the autogenerated script by hand before committing it -- Alembic's
autogenerate is a diffing aid, not a guarantee (e.g. it cannot always infer
column renames vs. drop+add, and comparing SQLite-reflected types against the
PostgreSQL-specific column types this project uses, e.g. `UUID`/`JSONB`, can
produce noisy false-positive diffs that don't apply to the real Postgres target).
